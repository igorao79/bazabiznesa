# DECISIONS.md — Ключевые архитектурные решения

## 1. Стек: Node.js + Express + React + PostgreSQL

Выбран JavaScript/TypeScript fullstack для единого языка на фронте и бэке. Express — минималистичный и гибкий фреймворк, не навязывает лишних абстракций. PostgreSQL — надёжная СУБД с хорошей поддержкой конкурентного доступа и транзакций, что критично для задачи с race condition.

## 2. Prisma ORM для работы с базой данных

Prisma обеспечивает типобезопасные запросы (TypeScript), автоматическую генерацию миграций и удобный seed-механизм. Встроенная поддержка optimistic concurrency control через `updateMany` с проверкой `version` — именно это используется для защиты от гонки.

## 3. Optimistic Concurrency Control (OCC) для защиты от гонки

Вместо пессимистичных блокировок (`SELECT FOR UPDATE`) выбран оптимистичный подход: поле `version` в таблице `requests`. При "Взять в работу" выполняется `updateMany` с условием `WHERE id = ? AND status = 'assigned' AND version = ?`. Если версия изменилась — `count = 0`, и возвращается `409 Conflict`. Это масштабируемый подход, не блокирующий другие операции чтения.

## 4. JWT для авторизации (без сессий)

Простая JWT-авторизация без серверных сессий. Токен содержит id, username, name, role. Для тестового проекта пароли хранятся в открытом виде в БД (в продакшне — bcrypt). Выбор JWT позволяет stateless-архитектуру, что упрощает горизонтальное масштабирование.

## 5. Разделение на слои: routes → services → prisma

- **Routes** — валидация HTTP-запросов, вызов сервисов, формирование ответов
- **Services** — бизнес-логика (проверка статусов, прав, OCC)
- **Prisma** — доступ к данным

Это даёт тестируемость (сервисы можно тестировать отдельно от HTTP), читаемость и возможность переиспользования логики.

## 6. Docker Compose с тремя сервисами

- `db` — PostgreSQL с healthcheck
- `backend` — Node.js Express API (порт 3001)
- `frontend` — Nginx с React SPA + reverse proxy к backend (порт 3000)

Nginx проксирует `/api/*` на backend, что позволяет фронтенду работать на том же домене без CORS-проблем в продакшне. Healthcheck для PostgreSQL гарантирует, что backend запустится только после готовности БД.

## 7. Дополнительные фичи: Audit Log, приоритеты, комментарии, поиск

- **Audit Log** — отдельная таблица `audit_logs` фиксирует каждое изменение статуса с указанием пользователя, старого и нового статуса. Это критичная фича для ремонтной службы (кто и когда что сделал).
- **Приоритеты** (low/normal/high/urgent) — позволяют диспетчеру сортировать и визуально выделять срочные заявки.
- **Комментарии** — диспетчер и мастер могут обмениваться заметками по заявке, не выходя из системы.
- **Полнотекстовый поиск** — поиск по клиенту, адресу, телефону, описанию через Prisma `contains` + `mode: insensitive`.
